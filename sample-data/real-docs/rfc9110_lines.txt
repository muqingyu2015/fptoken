 HTTP Semantics

Abstract

 The Hypertext Transfer Protocol (HTTP) is a stateless applicati
 level protocol for distributed, collaborative, hypertext inform
 systems. This document describes the overall architecture of HT
 establishes common terminology, and defines aspects of the prot
 that are shared by all versions. In this definition are core
 protocol elements, extensibility mechanisms, and the "http" and
 "https" Uniform Resource Identifier (URI) schemes.

 This document updates RFC 3864 and obsoletes RFCs 2818, 7231, 7
 7233, 7235, 7538, 7615, 7694, and portions of 7230.

1. Introduction

1.1. Purpose

 The Hypertext Transfer Protocol (HTTP) is a family of stateless
 application-level, request/response protocols that share a gene
 interface, extensible semantics, and self-descriptive messages 
 enable flexible interaction with network-based hypertext inform
 systems.

 HTTP hides the details of how a service is implemented by prese
 a uniform interface to clients that is independent of the types
 resources provided. Likewise, servers do not need to be aware o
 each client's purpose: a request can be considered in isolation
 rather than being associated with a specific type of client or 
 predetermined sequence of application steps. This allows genera
 purpose implementations to be used effectively in many differen
 contexts, reduces interaction complexity, and enables independe
 evolution over time.

 HTTP is also designed for use as an intermediation protocol, wh
 proxies and gateways can translate non-HTTP information systems
 a more generic interface.

 One consequence of this flexibility is that the protocol cannot
 defined in terms of what occurs behind the interface. Instead, 
 are limited to defining the syntax of communication, the intent
 received communication, and the expected behavior of recipients
 the communication is considered in isolation, then successful a
 ought to be reflected in corresponding changes to the observabl
 interface provided by servers. However, since multiple clients 
 act in parallel and perhaps at cross-purposes, we cannot requir
 such changes be observable beyond the scope of a single respons

1.2. History and Evolution

 HTTP has been the primary information transfer protocol for the
 Wide Web since its introduction in 1990. It began as a trivial
 mechanism for low-latency requests, with a single method (GET) 
 request transfer of a presumed hypertext document identified by
 given pathname. As the Web grew, HTTP was extended to enclose
 requests and responses within messages, transfer arbitrary data
 formats using MIME-like media types, and route requests through
 intermediaries. These protocols were eventually defined as HTTP
 and HTTP/1.0 (see [HTTP/1.0]).

 HTTP/1.1 was designed to refine the protocol's features while
 retaining compatibility with the existing text-based messaging
 syntax, improving its interoperability, scalability, and robust
 across the Internet. This included length-based data delimiters
 both fixed and dynamic (chunked) content, a consistent framewor
 content negotiation, opaque validators for conditional requests
 cache controls for better cache consistency, range requests for
 partial updates, and default persistent connections. HTTP/1.1 w
 introduced in 1995 and published on the Standards Track in 1997
 [RFC2068], revised in 1999 [RFC2616], and revised again in 2014
 ([RFC7230] through [RFC7235]).

 HTTP/2 ([HTTP/2]) introduced a multiplexed session layer on top
 the existing TLS and TCP protocols for exchanging concurrent HT
 messages with efficient field compression and server push. HTTP
 ([HTTP/3]) provides greater independence for concurrent message
 using QUIC as a secure multiplexed transport over UDP instead o

 All three major versions of HTTP rely on the semantics defined 
 this document. They have not obsoleted each other because each 
 has specific benefits and limitations depending on the context 
 use. Implementations are expected to choose the most appropriat
 transport and messaging syntax for their particular context.

 This revision of HTTP separates the definition of semantics (th
 document) and caching ([CACHING]) from the current HTTP/1.1 mes
 syntax ([HTTP/1.1]) to allow each major protocol version to pro
 independently while referring to the same core semantics.

1.3. Core Semantics

 HTTP provides a uniform interface for interacting with a resour
 (Section 3.1) -- regardless of its type, nature, or implementat
 by sending messages that manipulate or transfer representations
 (Section 3.2).

 Each message is either a request or a response. A client constr
 request messages that communicate its intentions and routes tho
 messages toward an identified origin server. A server listens f
 requests, parses each message received, interprets the message
 semantics in relation to the identified target resource, and re
 to that request with one or more response messages. The client
 examines received responses to see if its intentions were carri
 out, determining what to do next based on the status codes and
 content received.

 HTTP semantics include the intentions defined by each request m
 (Section 9), extensions to those semantics that might be descri
 request header fields, status codes that describe the response
 (Section 15), and other control data and resource metadata that
 be given in response fields.

 Semantics also include representation metadata that describe ho
 content is intended to be interpreted by a recipient, request h
 fields that might influence content selection, and the various
 selection algorithms that are collectively referred to as "cont
 negotiation" (Section 12).

2. Conformance

2.1. Syntax Notation

 This specification uses the Augmented Backus-Naur Form (ABNF)
 notation of [RFC5234], extended with the notation for case-
 sensitivity in strings defined in [RFC7405].

 It also uses a list extension, defined in Section 5.6.1, that a
 for compact definition of comma-separated lists using a "#" ope
 (similar to how the "*" operator indicates repetition). Appendi
 shows the collected grammar with all list operators expanded to
 standard ABNF notation.

 As a convention, ABNF rule names prefixed with "obs-" denote ob
 grammar rules that appear for historical reasons.

 The following core rules are included by reference, as defined 
 Appendix B.1 of [RFC5234]: ALPHA (letters), CR (carriage return
 CRLF (CR LF), CTL (controls), DIGIT (decimal 0-9), DQUOTE (doub
 quote), HEXDIG (hexadecimal 0-9/A-F/a-f), HTAB (horizontal tab)
 (line feed), OCTET (any 8-bit sequence of data), SP (space), an
 VCHAR (any visible US-ASCII character).

 Section 5.6 defines some generic syntactic components for field
 values.

 This specification uses the terms "character", "character encod
 scheme", "charset", and "protocol element" as they are defined 
 [RFC6365].

2.2. Requirements Notation

 The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL N
 "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY"
 "OPTIONAL" in this document are to be interpreted as described 
 BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in 
 capitals, as shown here.

 This specification targets conformance criteria according to th
 of a participant in HTTP communication. Hence, requirements are
 placed on senders, recipients, clients, servers, user agents,
 intermediaries, origin servers, proxies, gateways, or caches,
 depending on what behavior is being constrained by the requirem
 Additional requirements are placed on implementations, resource
 owners, and protocol element registrations when they apply beyo
 scope of a single communication.

 The verb "generate" is used instead of "send" where a requireme
 applies only to implementations that create the protocol elemen
 rather than an implementation that forwards a received element
 downstream.

 An implementation is considered conformant if it complies with 
 the requirements associated with the roles it partakes in HTTP.

 A sender MUST NOT generate protocol elements that do not match 
 grammar defined by the corresponding ABNF rules. Within a given
 message, a sender MUST NOT generate protocol elements or syntax
 alternatives that are only allowed to be generated by participa
 other roles (i.e., a role that the sender does not have for tha
 message).

 Conformance to HTTP includes both conformance to the particular
 messaging syntax of the protocol version in use and conformance
 the semantics of protocol elements sent. For example, a client 
 claims conformance to HTTP/1.1 but fails to recognize the featu
 required of HTTP/1.1 recipients will fail to interoperate with
 servers that adjust their responses in accordance with those cl
 Features that reflect user choices, such as content negotiation
 user-selected extensions, can impact application behavior beyon
 protocol stream; sending protocol elements that inaccurately re
 a user's choices will confuse the user and inhibit choice.

 When an implementation fails semantic conformance, recipients o
 implementation's messages will eventually develop workarounds t
 adjust their behavior accordingly. A recipient MAY employ such
 workarounds while remaining conformant to this protocol if the
 workarounds are limited to the implementations at fault. For
 example, servers often scan portions of the User-Agent field va
 and user agents often scan the Server field value, to adjust th
 own behavior with respect to known bugs or poorly chosen defaul

2.3. Length Requirements

 A recipient SHOULD parse a received protocol element defensivel
 with only marginal expectations that the element will conform t
 ABNF grammar and fit within a reasonable buffer size.

 HTTP does not have specific length limitations for many of its
 protocol elements because the lengths that might be appropriate
 vary widely, depending on the deployment context and purpose of
 implementation. Hence, interoperability between senders and
 recipients depends on shared expectations regarding what is a
 reasonable length for each protocol element. Furthermore, what 
 commonly understood to be a reasonable length for some protocol
 elements has changed over the course of the past three decades 
 HTTP use and is expected to continue changing in the future.

 At a minimum, a recipient MUST be able to parse and process pro
 element lengths that are at least as long as the values that it
 generates for those same protocol elements in other messages. F
 example, an origin server that publishes very long URI referenc
 its own resources needs to be able to parse and process those s
 references when received as a target URI.

 Many received protocol elements are only parsed to the extent
 necessary to identify and forward that element downstream. For
 example, an intermediary might parse a received field into its 
 name and field value components, but then forward the field wit
 further parsing inside the field value.

2.4. Error Handling

 A recipient MUST interpret a received protocol element accordin
 the semantics defined for it by this specification, including
 extensions to this specification, unless the recipient has dete
 (through experience or configuration) that the sender incorrect
 implements what is implied by those semantics. For example, an
 origin server might disregard the contents of a received
 Accept-Encoding header field if inspection of the User-Agent he
 field indicates a specific implementation version that is known
 fail on receipt of certain content codings.

 Unless noted otherwise, a recipient MAY attempt to recover a us
 protocol element from an invalid construct. HTTP does not defin
 specific error handling mechanisms except when they have a dire
 impact on security, since different applications of the protoco
 require different error handling strategies. For example, a Web
 browser might wish to transparently recover from a response whe
 Location header field doesn't parse according to the ABNF, wher
 systems control client might consider any form of error recover
 be dangerous.

 Some requests can be automatically retried by a client in the e
 of an underlying connection failure, as described in Section 9.

2.5. Protocol Version

 HTTP's version number consists of two decimal digits separated 
 "." (period or decimal point). The first digit (major version)
 indicates the messaging syntax, whereas the second digit (minor
 version) indicates the highest minor version within that major
 version to which the sender is conformant (able to understand f
 future communication).

 While HTTP's core semantics don't change between protocol versi
 their expression "on the wire" can change, and so the HTTP vers
 number changes when incompatible changes are made to the wire f
 Additionally, HTTP allows incremental, backwards-compatible cha
 to be made to the protocol without changing its version through
 use of defined extension points (Section 16).

 The protocol version as a whole indicates the sender's conforma
 with the set of requirements laid out in that version's corresp
 specification(s). For example, the version "HTTP/1.1" is define
 the combined specifications of this document, "HTTP Caching"
 [CACHING], and "HTTP/1.1" [HTTP/1.1].

 HTTP's major version number is incremented when an incompatible
 message syntax is introduced. The minor number is incremented w
 changes made to the protocol have the effect of adding to the m
 semantics or implying additional capabilities of the sender.

 The minor version advertises the sender's communication capabil
 even when the sender is only using a backwards-compatible subse
 the protocol, thereby letting the recipient know that more adva
 features can be used in response (by servers) or in future requ
 (by clients).

 When a major version of HTTP does not define any minor versions
 minor version "0" is implied. The "0" is used when referring to
 protocol within elements that require a minor version identifie

3. Terminology and Core Concepts

 HTTP was created for the World Wide Web (WWW) architecture and 
 evolved over time to support the scalability needs of a worldwi
 hypertext system. Much of that architecture is reflected in the
 terminology used to define HTTP.

3.1. Resources

 The target of an HTTP request is called a "resource". HTTP does
 limit the nature of a resource; it merely defines an interface 
 might be used to interact with resources. Most resources are
 identified by a Uniform Resource Identifier (URI), as described
 Section 4.

 One design goal of HTTP is to separate resource identification 
 request semantics, which is made possible by vesting the reques
 semantics in the request method (Section 9) and a few request-
 modifying header fields. A resource cannot treat a request in a
 manner inconsistent with the semantics of the method of the req
 For example, though the URI of a resource might imply semantics
 are not safe, a client can expect the resource to avoid actions
 are unsafe when processing a request with a safe method (see
 Section 9.2.1).

 HTTP relies upon the Uniform Resource Identifier (URI) standard
 to indicate the target resource (Section 7.1) and relationships
 between resources.

3.2. Representations

 A "representation" is information that is intended to reflect a
 current, or desired state of a given resource, in a format that
 be readily communicated via the protocol. A representation cons
 of a set of representation metadata and a potentially unbounded
 stream of representation data (Section 8).

 HTTP allows "information hiding" behind its uniform interface b
 defining communication with respect to a transferable represent
 of the resource state, rather than transferring the resource it
 This allows the resource identified by a URI to be anything,
 including temporal functions like "the current weather in Lagun
 Beach", while potentially providing information that represents
 resource at the time a message is generated [REST].

 The uniform interface is similar to a window through which one 
 observe and act upon a thing only through the communication of
 messages to an independent actor on the other side. A shared
 abstraction is needed to represent ("take the place of") the cu
 or desired state of that thing in our communications. When a
 representation is hypertext, it can provide both a representati
 the resource state and processing instructions that help guide 
 recipient's future interactions.

 A target resource might be provided with, or be capable of
 generating, multiple representations that are each intended to
 reflect the resource's current state. An algorithm, usually bas
 content negotiation (Section 12), would be used to select one o
 those representations as being most applicable to a given reque
 This "selected representation" provides the data and metadata f
 evaluating conditional requests (Section 13) and constructing t
 content for 200 (OK), 206 (Partial Content), and 304 (Not Modif
 responses to GET (Section 9.3.1).

3.3. Connections, Clients, and Servers

 HTTP is a client/server protocol that operates over a reliable
 transport- or session-layer "connection".

 An HTTP "client" is a program that establishes a connection to 
 server for the purpose of sending one or more HTTP requests. An
 "server" is a program that accepts connections in order to serv
 HTTP requests by sending HTTP responses.

 The terms client and server refer only to the roles that these

 a connection. Many implementations depend on HTTP's stateless d

 As a result, a server MUST NOT assume that two requests on the

6. Message Abstraction

 Each major version of HTTP defines its own syntax for communica
 messages. This section defines an abstract data type for HTTP
 messages based on a generalization of those message characteris
 common structure, and capacity for conveying semantics. This
 abstraction is used to define requirements on senders and recip
 that are independent of the HTTP version, such that a message i
 version can be relayed through other versions without changing 
 meaning.

 A "message" consists of the following:

 * control data to describe and route the message,

 * a headers lookup table of name/value pairs for extending that
 control data and conveying additional information about the
 sender, message, content, or context,

 * a potentially unbounded stream of content, and

 * a trailers lookup table of name/value pairs for communicating
 information obtained while sending the content.

 Framing and control data is sent first, followed by a header se
 containing fields for the headers table. When a message include
 content, the content is sent after the header section, potentia
 followed by a trailer section that might contain fields for the
 trailers table.

 Messages are expected to be processed as a stream, wherein the
 purpose of that stream and its continued processing is revealed
 being read. Hence, control data describes what the recipient ne
 to know immediately, header fields describe what needs to be kn
 before receiving content, the content (when present) presumably
 contains what the recipient wants or needs to fulfill the messa
 semantics, and trailer fields provide optional metadata that wa
 unknown prior to sending the content.

 Messages are intended to be "self-descriptive": everything a
 recipient needs to know about the message can be determined by
 looking at the message itself, after decoding or reconstituting
 that have been compressed or elided in transit, without requiri
 understanding of the sender's current application state (establ
 via prior messages). However, a client MUST retain knowledge of
 request when parsing, interpreting, or caching a corresponding
 response. For example, responses to the HEAD method look just l
 the beginning of a response to GET but cannot be parsed in the 
 manner.

 Note that this message abstraction is a generalization across m
 versions of HTTP, including features that might not be found in
 versions. For example, trailers were introduced within the HTTP
 chunked transfer coding as a trailer section after the content.
 equivalent feature is present in HTTP/2 and HTTP/3 within the h
 block that terminates each stream.

6.1. Framing and Completeness

 Message framing indicates how each message begins and ends, suc
 each message can be distinguished from other messages or noise 
 same connection. Each major version of HTTP defines its own fra
 mechanism.

 HTTP/0.9 and early deployments of HTTP/1.0 used closure of the
 underlying connection to end a response. For backwards

 However, implicit framing can fail to distinguish an incomplete
 response if the connection closes early. For that reason, almos

 A message is considered "complete" when all of the octets indic

 (but separate from) the header fields. Additional processing
 expectations, if any, can be defined within the field specifica
 for a field intended for use in trailers.

6.6. Message Metadata

 Fields that describe the message itself, such as when and how t
 message has been generated, can appear in both requests and
 responses.

6.6.1. Date

 The "Date" header field represents the date and time at which t
 message was originated, having the same semantics as the Origin
 Date Field (orig-date) defined in Section 3.6.1 of [RFC5322]. T
 field value is an HTTP-date, as defined in Section 5.6.7.

 Date = HTTP-date

 An example is

 Date: Tue, 15 Nov 1994 08:12:31 GMT

 A sender that generates a Date header field SHOULD generate its
 value as the best available approximation of the date and time 
 message generation. In theory, the date ought to represent the
 moment just before generating the message content. In practice,
 sender can generate the date value at any time during message
 origination.

 An origin server with a clock (as defined in Section 5.6.7) MUS
 generate a Date header field in all 2xx (Successful), 3xx
 (Redirection), and 4xx (Client Error) responses, and MAY genera
 Date header field in 1xx (Informational) and 5xx (Server Error)
 responses.

 An origin server without a clock MUST NOT generate a Date heade
 field.

 A recipient with a clock that receives a response message witho
 Date header field MUST record the time it was received and appe
 corresponding Date header field to the message's header section
 is cached or forwarded downstream.

 A recipient with a clock that receives a response with an inval
 Date header field value MAY replace that value with the time th
 response was received.

 A user agent MAY send a Date header field in a request, though
 generally will not do so unless it is believed to convey useful
 information to the server. For example, custom applications of 
 might convey a Date if the server is expected to adjust its
 interpretation of the user's request based on differences betwe
 user agent and server clocks.

6.6.2. Trailer

 The "Trailer" header field provides a list of field names that 
 sender anticipates sending as trailer fields within that messag
 This allows a recipient to prepare for receipt of the indicated
 metadata before it starts processing the content.

 Trailer = #field-name

 For example, a sender might indicate that a signature will be
 computed as the content is being streamed and provide the final
 signature as a trailer field. This allows a recipient to perfor
 same check on the fly as it receives the content.

 A sender that intends to generate one or more trailer fields in
 message SHOULD generate a Trailer header field in the header se
 of that message to indicate which fields might be present in th
 trailers.

 If an intermediary discards the trailer section in transit, the
 Trailer field could provide a hint of what metadata was lost, t
 there is no guarantee that a sender of Trailer will always foll
 through by sending the named fields.

7. Routing HTTP Messages

 HTTP request message routing is determined by each client based
 the target resource, the client's proxy configuration, and
 establishment or reuse of an inbound connection. The correspond
 response routing follows the same connection chain back to the
 client.
