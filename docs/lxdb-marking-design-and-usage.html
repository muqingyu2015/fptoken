LXDB 打标功能 — 设计与使用说明

本文档整合 SQL 侧配置与 Java 引擎实现（包 cn.lucene.lxdb.api.fdw.parser.params.parser.fn_json_impl.format.marking）。

一、开发背景

场景一：批量数据打标
需要一次性对大量数据打标（例如将号码 10086 标记为公共号码）。数据量可达数十亿行，若通过频繁 UPDATE 写回大表，数据库难以承受。

场景二：字典值获取
需要将原表中的字典 ID 与字典表关联，得到可读的业务含义或展示值。

二、业界常规做法与 LXDB 思路

常规做法是多表 JOIN。但在数亿到数百亿行规模下，大表与字典表直接关联往往使查询不可接受。

LXDB 做法：通过系统规则表 lxdb_sys.join_marking_rule 描述「原表字段 ↔ 规则 SQL 产出」的关系，引擎在计算层将规则数据载入并建立内存侧索引，扫描原表行时用索引快速判定命中哪些 mark_name，以虚拟列（text[] 等）形式返回，避免对大表做批量物理更新。

关系示意：[原表] ↔ [lxdb_sys.join_marking_rule] ↔ [规则 SQL 指向的打标/字典数据]

  1. 准备好原表，在要打标的列上使用 OPTIONS(markrule '规则名') 绑定规则名。
  2. 在 join_marking_rule 中按 rule_name 找到对应 marking_sql。
  3. 引擎执行 marking_sql，将得到的 mark_name + mark_filter（JSON）建成内存索引。
  4. 读取原表每一行时，用当前行的 DocValues 与规则索引匹配，得到打标结果列表。

三、旧环境：系统规则表 DDL

create schema if not exists lxdb_sys;
create table lxdb_sys.join_marking_rule(
    rule_name text,
    marking_sql text,
    memo text
);

CREATE INDEX lxdb_sys_join_marking_rule on lxdb_sys.join_marking_rule USING BRIN(rule_name);
select create_distributed_table('lxdb_sys.join_marking_rule', 'rule_name','hash','default',2);

四、使用说明（SQL）

4.1 原始表与打标列

CREATE FOREIGN TABLE big_table_marking_example (
ukey text OPTIONS(key 'unique'),
i1 int,
i2 int,
txt1 text,
txt2 text,
marking1 text[] options(markrule 'common_example'),
marking2 text[] options(markrule 'rule1_eq_join'),
marking3 text[] options(markrule 'rule2_range_join'),

select create_distributed_table('big_table_marking_example ', 'ukey','hash','default',2);

4.2 常规打标：显式 JSON 规则表

create table marking_example1(
 mark_name text ,
 mark_filter json
);
select create_distributed_table('marking_example1', 'mark_name','hash','default',2);

-- 等值打标
insert into marking_example1 (mark_name,mark_filter) values('客服电话','{"eq" : [{"f" : "txt1", "v" : ["10086"]}]}');
insert into marking_example1 (mark_name,mark_filter) values('号码查询','{"eq" : [{"f" : "txt1", "v" : ["114"]}]}');

-- 范围打标
insert into marking_example1 (mark_name,mark_filter) values('VIP客户','{"range" : [{"f" : "i1", "v" : [500, 1000000]}]}');
insert into marking_example1 (mark_name,mark_filter) values('头部客户','{"range" : [{"f" : "i1", "v" : [1000, 1000000]}]}');
insert into marking_example1 (mark_name,mark_filter) values('普通客户','{"range" : [{"f" : "i1", "v" : [0, 500]}]}');

delete from lxdb_sys.join_marking_rule where rule_name='common_example' ;
insert into lxdb_sys.join_marking_rule(rule_name,memo ,marking_sql)
VALUES (
'common_example','常规打标例子',
$$select mark_name,mark_filter from marking_example1 $$
);

查询示例结果（示意）：

select * from big_table_marking_example;
-- marking1 列将按规则聚合多个命中的 mark_name

4.3 高级打标：用 SQL 动态拼装 mark_filter

适用于字典表结构已固定、不便直接存 JSON 的场景：在 marking_sql 中用 json_build_object 等生成与常规打标相同形态的 mark_filter。

等值关联示例（号码）

CREATE FOREIGN TABLE phone_num_marking(
ukey text OPTIONS(key 'unique'),
label text,
phone_num text
) SERVER lxdb options(store 'ids');

select create_distributed_table('phone_num_marking', 'ukey','hash','default',2);

insert into phone_num_marking(ukey,label, phone_num) VALUES ('001','客服电话','10086');
insert into phone_num_marking(ukey,label, phone_num) VALUES ('002','号码查询','114');

delete from lxdb_sys.join_marking_rule where rule_name='rule1_eq_join' ;
insert into lxdb_sys.join_marking_rule(rule_name,memo ,marking_sql)
VALUES (
'rule1_eq_join','号码打标例子',
$$select 
    label as mark_name,
    json_build_object(
        'eq', json_build_array(
            json_build_object('f', 'txt1', 'v', json_build_array(phone_num))
        )
    ) as mark_filter 
from phone_num_marking$$
);

范围关联示例

CREATE FOREIGN TABLE range_marking(
ukey text OPTIONS(key 'unique'),
label text,
v_from float8,
v_to float8
) SERVER lxdb options(store 'ids');

select create_distributed_table('range_marking', 'ukey','hash','default',2);

insert into range_marking(ukey,label, v_from,v_to) VALUES ('003','普通客户',0,500);
insert into range_marking(ukey,label, v_from,v_to) VALUES ('004','VIP客户',500,1000000);
insert into range_marking(ukey,label, v_from,v_to) VALUES ('005','头部客户',1000,1000000);

insert into lxdb_sys.join_marking_rule(rule_name,memo ,marking_sql)
VALUES ('rule2_range_join','范围匹配例子',
$$select 
    label as mark_name,
    json_build_object(
        'range', json_build_array(
            json_build_object('f', 'i1', 'v', json_build_array(v_from,v_to))
        )
    ) as mark_filter 
from range_marking$$);

select * from big_table_marking_example;
select marking2,count(*) from big_table_marking_example group by marking2;
select * from big_table_marking_example where marking2[0] like '%客户%';

五、mark_filter JSON 语义（引擎支持）

规则引擎在建索引与按行匹配两侧使用同一套结构解析（见 JoinMarkingJsonParser、RuleBasedLuceneIndexerDocumentBuilder、RuleBasedLuceneIndexerOrdReader）。

键含义说明
and逻辑与子规则数组，全部满足。
or逻辑或子规则数组，任一满足。
eq等值每项：{"f":"列名","v":["值",...]}；数值列会尝试规范化格式以匹配索引项。
range闭区间范围v 为两个边界；数值用前缀编码范围查询，否则用字符串 srange@... 边界字段。
rangeNoEq开区间范围边界不包含端点（数值与字符串两套路径）。
contains子串/包含语义匹配时使用 LxdbPartOfQuery。
partof反向包含语义匹配时使用 LxdbContainsQuery（规则值与字段值的「部分」关系与 contains 方向不同）。

JoinMarkingJsonParser.getValues 允许 v 等为 JSONArray、单个 JSONObject、JSON 字符串或逗号分隔列表，便于与不同 SQL 生成方式对接。

六、Java 实现架构（format.marking 包）

该包实现「从 PG 拉规则 → 建 Lucene 内存索引 → 对数据行求多值打标」以及「查询侧按打标值反推 Lucene 过滤条件」。

类职责
JoinMarkingJsonParser 从 mark_filter JSONObject 中按 key 取出条件数组，兼容多种 JSON 形态。
MarkingTempDirectory 在 LXDB_STORE_PATH 下创建规则索引临时目录（/lxfs/lxtmp/marking/{lxdb.dn.mark}/），并挂定时清理。
PeriodicMarkingIndexLoader 按 rule_name 从 lxdb_sys.join_marking_rule 读 marking_sql 并执行，将行集写入 RuleBasedLuceneIndexer；后台线程按数据量调整周期重新 loadIndexData；24 小时无活跃则从缓存移除。
RuleBasedLuceneIndexer 维护单段合并后的 Lucene 索引、ruleMap（规则 JSON）、涉及字段集 fieldSet；ORD_STEP_SIZE 将 (叶子序号, doc) 编码为长整型 ord。
RuleBasedLuceneIndexerDocumentBuilder 将每条规则的 mark_filter 展开为多个索引字段（如 eq@列@路径、range@...），并写入 mark_name 的 BinaryDocValues。
RuleBasedLuceneIndexerOrdReader 对给定 docId，从原表各列 DocValues 取值，拼装与建索引时对称的 Lucene Query，在规则索引中搜索命中文档，返回 ord 列表。
DocumentMarker 实现 lxdb_fn_json_format：参数 JSON 含 fn_name=join_marking、rule_name；通过 RULE_CACHE 复用 PeriodicMarkingIndexLoader；getDocReader 为规则涉及列构造 CollectPairDocValuesOrds 并封装为 JoinMarkingDocValues。
MarkingFilterQueryBuilder 供 LxdbQueryParser 使用：当 WHERE 对打标列过滤时，根据列对应的 rule_name 再查 PG，把符合条件的 mark_filter 转成原表字段上的 Lucene Query（与索引侧语义对齐），实现「按标签筛行」。

6.1 运行时数据流（读路径）

SQL 列 OPTIONS(markrule) → LxdbTableInfo.field2Marking → MarkingFilterQueryBuilder.initializeMarkingRules 注册 format_fn（DocumentMarker） → DocumentMarker.loadRuleIndexer(rule_name) → PeriodicMarkingIndexLoader：JDBC 执行 marking_sql，RuleBasedLuceneIndexer 建索引 → JoinMarkingDocValues.getOrds(docid)：用本行各字段 DocValues + 规则索引检索 → 多个 mark_name ord → 列值解码为 text[] 等多值展示

6.2 环境与连接

  • 加载规则数据使用环境变量 LXDB_CN_HOSTNAME（及固定端口 5432、库 lxdb 等，以代码为准）。
  • MarkingFilterQueryBuilder.fetchMarkingRules 与 PeriodicMarkingIndexLoader.fetchMarkingData 均依赖上述连接读取 join_marking_rule 及规则 SQL 结果。
  • 规则索引落在磁盘临时目录，非纯 RAMDirectory；大量规则时注意磁盘与清理策略。

七、小结

  • 业务侧：用 markrule 绑定规则名；用 marking_sql 产出 mark_name + mark_filter，既可手写 JSON 表，也可用 SQL 动态生成。
  • 引擎侧：format.marking 将规则集编译为 Lucene 索引，按行用 DocValues 做匹配，避免大表与字典的在线 JOIN；MarkingFilterQueryBuilder 支持按标签反向过滤。
  • 扩展：在 JSON 中组合 and/or 与 eq/range/contains/partof 等即可表达更复杂的打标条件（需与建索引、读匹配实现保持一致）。
