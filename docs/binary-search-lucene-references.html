12.2 Zoekt（Trigram 代码搜索）

要点

基本原理/思路： 对每个 trigram 不仅存「哪些文档包含」，还存在文档内的出现位置（偏移）。子串查询时取查询串中的若干 trigram（如首、尾），在索引中取交并检查它们在同一文档内的相对距离是否与查询子串一致，从而在不读全文的情况下筛出候选，再做验证。仅访问少量 posting list，适合 SSD 与低延迟。

  • 场景： 面向 Git 仓库的全文/子串代码搜索，目标单机 SSD 上亚 50ms 级延迟（如 Android 约 2G 文本）。
  • 做法： 位置化 trigram（positional trigram）：每个 trigram 存其在文件内的偏移。子串查询时提取查询中的 trigram（如首尾或关键几个），要求它们在文档中按正确距离出现，再验证。仅需少量 posting list 相交，适合 SSD。
  • 优化： 将 n-gram 从 map 改为两层 slice 索引，显著降低内存（如 14.5GB→3.5GB）并加快加载。

创新点： 位置化 trigram：存偏移而非仅 docId，用「首尾 gram + 文档内距离」在不读全文的情况下筛候选，仅访问少量 posting list，适合 SSD 与亚 50ms 延迟目标；两层 slice 索引替代 map，大幅省内存。

可借鉴的地方： 用「距离约束」减少候选集与验证量；少量 list 交即可得到高精度候选的思路；内存紧凑的索引结构（slice/扁平化）可类比本文 DocList 分块与压缩。

12.3 Milvus NGRAM 索引

要点

基本原理/思路： 建索引时用滑动窗口把文本切成若干长度的 n-gram（min_gram～max_gram），每个 gram 映射到「包含该 gram 的主键/文档 ID」，写成倒排；查询时把查询串同样切 n-gram，在倒排里取交得到候选 ID，再对候选做精确的 LIKE 或字符串匹配。本质是「gram 倒排 + 候选再验证」，与经典全文子串加速一致。

  • 场景： 向量库 Milvus 对 VARCHAR 字段的 LIKE '%xxx%' 类查询加速。
  • 做法： 建索引时用滑动窗口将文本切为 n-gram（可配置 min_gram / max_gram），存入倒排索引（gram → 文档 ID）；查询时用查询串的 n-gram 过滤出候选集，再精确匹配。支持前缀、后缀、中缀子串。
  • 与本文的对应： 思路一致：「滑动窗口 → token → 倒排 → 候选 → 验证」。本文用 64B 定长、BitSet 做块内过滤，Milvus 用变长 n-gram + 倒排。

创新点： 在向量库内原生支持 VARCHAR 的 LIKE 子串加速，min_gram/max_gram 可配，与向量检索混合使用；建索引与查询两阶段清晰，文档与 API 完善，便于工程落地。

可借鉴的地方： 滑动窗口 + 倒排 + 候选再精确匹配的完整链路；min_gram/max_gram 参数对召回与索引体积的权衡；若本文将来支持变长块，可参考其 gram 长度配置策略。

12.4 Elasticsearch / Lucene 与「二进制」

要点

基本原理/思路： binary 字段在写入时只做 Base64 解码并按原样存入 stored 流，不建倒排、不建词典，因此不能按内容检索。要对「二进制文档」做搜索时，用 Tika 或 Data Extraction 等把 PDF/Office 等解析成文本，再对文本走标准分词与倒排索引；检索对象是「提取出的文本」而非原始字节流。

  • Binary 字段： Elasticsearch 的 binary 类型默认不可检索，仅可存储（Base64 解码后存）；若要对二进制内容做全文检索，需先提取文本再索引。
  • 二进制文档（PDF/Office）： 通过 Apache Tika 或 Elastic 的 Data Extraction Service 等先提取文本，再对文本建全文索引，而不是对原始字节建子串索引。
  • BBQ： Lucene/Elasticsearch 中的 Better Binary Quantization 是针对向量的二值量化，与「二进制流子串检索」无直接关系。

创新点： 将「二进制文档检索」明确拆成「存 binary」与「提文本再全文索引」两条路径，Tika/Data Extraction 生态成熟，适合文档类二进制；binary 字段语义简单、不参与检索，避免误用。

可借鉴的地方： 明确「只存不检」与「可检索」的边界，本文 _bin 扩展正好补足「可检」一侧；文档抽取流水线与本文「先处理再索引」的衔接方式可参考；若需同时支持「提文本」与「原始 byte[] 子串」，可复用其 ingest 与字段设计思路。

结论： 业界通用栈中通常不对原始二进制做类 LIKE 的子串索引，要么只存不查，要么先转文本再全文检索。本文方案填补的是「对原始 byte[] 做高召回子串检索」这一缺口。

参考链接： Elasticsearch binary field；Binary document ingestion (Elasticsearch Labs)

12.5 ClickHouse Data Skipping Index

要点

基本原理/思路： 数据按块（granule）存储，每块维护轻量级元数据：minmax 存该块内某列的最小/最大值；bloom_filter 存该块内值的集合的近似；ngrambf_v1 对该块内字符串做 n-gram 切分后放入 Bloom。查询时先查这些元数据，若块不可能命中（如查询值不在 minmax 范围、或 Bloom 判「一定不存在」）则整块跳过，只读可能命中的块，从而减少 I/O。

  • 场景： 按块（granularity）跳过不相关数据，减少扫描量；适用于范围、等值、子串/包含等条件。
  • 类型： minmax（块内最小/最大值）、set(N)（块内取值集合）、bloom_filter、ngrambf_v1 / tokenbf_v1（基于 n-gram 或 token 的 Bloom）、text（分词倒排）等。
  • 与本文的对应： 本文在 3.3.2 中提到的「段/块内 min-max、首字节集、Bloom、q-gram」等二次过滤，与 ClickHouse 的块级 minmax/Bloom/ngram 思路一致，都是先块级过滤再读细粒度数据。

创新点： 多种跳过索引类型（minmax、set、bloom_filter、ngrambf_v1、tokenbf_v1）与 granularity 解耦，按列与查询形态选配；块级元数据轻量，I/O 节约明显，文档与生产实践丰富。

可借鉴的地方： 块级 minmax/Bloom/ngram 与本文 3.3.2 段级二次过滤直接对应；granularity 与块大小的取舍；ngrambf 参数（长度、Bloom 大小）可参考用于本文 q-gram 或块内过滤设计。

12.6 Bit Matrix 子串搜索（Baeza-Yates / Richard Startin）

要点

基本原理/思路： 预处理阶段为 pattern 建 256 个 bitmask（每字节值一个）：mask[b] 的第 i 位为 1 表示 pattern[i]=b。扫描时维护一个状态字（初始 0）：每读一字节 c，状态左移一位并或 1，再与 mask[c] 做与；若状态的「第 pattern 长度 位」为 1，则当前窗口与 pattern 匹配。全程移位+与，无分支，pattern 长度受字长限制（如 ≤64）。

  • 思路： 为待查子串（pattern）构建 256×「位置」的位矩阵：每个字节值对应一个 bitmask，表示该字节在 pattern 中出现的所有位置。扫描文本时用单个 long 做「移位 + 与」更新状态，当高位被置 1 时表示匹配。天然分支无关，适合 SIMD/吞吐优化。
  • 限制： pattern 长度不能超过 64（因用 64-bit 字跟踪位置）；更长模式需分块或换算法。
  • 与本文的对应： 本文用「64B 窗口」与「256×3 BitSet」做候选过滤，Bit Matrix 用 256 个 mask 做单模式扫描；若把「查询块」视为固定 64B pattern，两者在「按字节查表 + 位运算」上有相似性。Sparse 版本只存 pattern 中实际出现的字节对应 mask，节省空间。

创新点： 用 256×位置 bitmask 与单字长状态实现子串匹配，全程移位+与、无分支，适合 SIMD/高吞吐；Sparse 变体只存出现字节的 mask，省空间且缓存友好；≤64 字节的约束与本文 64B 块天然契合。

可借鉴的地方： 按字节查表 + 位运算的过滤思路与本文 BitSet/256 表一致；精确比对阶段可考虑用 Bit Matrix 做单块（≤64B）扫描；Sparse 思路可用于减少本文 256 表中未出现字节的存储；SIMD 化时可参考其无分支设计。

参考链接： Sparse Bit Matrix Substring Search（Richard Startin）；Baeza-Yates, “A New Approach to Text Searching”（论文）。

12.7 关系库中的 BINARY / BYTEA 子串与前缀

要点

基本原理/思路： 二进制列按字节存，LIKE 通过逐字节比较实现；没有对「任意位置子串」建专用倒排，因此中缀匹配多需顺序扫或全表扫。为加速前缀查询，可对列建 B-tree（按字节序），或对 substring(列,1,N) 建函数索引，这样「前 N 字节等于某值」可走索引；PG 还可用 v >= x AND v < y 做前缀区间查询。

  • PostgreSQL： bytea 支持 LIKE，前缀查询可用 WHERE v LIKE E'\\x12\\x34%'；为优化可对前缀建索引，或用区间写法 v >= x AND v < y；大列可对 substring(data,1,N) 建函数索引。
  • MariaDB / MySQL： BINARY 类型支持 LIKE 子串匹配，区分大小写；索引对前缀有效。
  • 与本文的对应： 关系库侧重「单列前缀/等值」和简单子串，不做「全量二进制流、任意位置子串、高召回零漏检」的专用索引；本文的 64B 滑动窗口 + 块内 BitSet + 精确比对是针对后者的设计。

创新点： 在通用 SQL 下支持 bytea/BINARY 的 LIKE 与前缀；PG 的 substring 函数索引、区间写法 v≥x AND v<y 提供可组合的优化手段；实现简单、与现有 SQL 生态兼容。

可借鉴的地方： 前缀/等值场景下的 B-tree 与函数索引思路；若本文将来暴露「前缀查询」API，可参考其索引与区间查询设计；作为「无专用二进制子串索引」的对照，明确本文方案的差异化。
