FrequentItemsetMiner：是「全排列」吗？性能与业界做法

面向本仓库 cn.lxdb.plugins.muqingyu.fptoken.miner.FrequentItemsetMiner 的阅读笔记与工程扩展建议。

核心结论：当前实现是「在频繁 1-项集上，按递增下标做 DFS 组合扩展 + tidset 交集剪枝」，不是对全体词做全排列。每个项集在规范序下只枚举一次。

1. 源码在做什么（为何容易误读成排列）

外层 for (i = start; i < frequentTermIds.size(); i++) 与递归里传入的 i + 1，等价于：从已选前缀后面继续选更大的 termId，保证 termId 严格递增。

  • 排列关心顺序，例如 {a,b} 与 {b,a} 算两种；若用全排列会重复计数同一项集。
  • 组合不关心顺序，规范写法是递增序列，本实现正是这种「组合枚举」。

支持度通过 BitSet 交集得到：T(X ∪ {t}) = T(X) ∩ T({t})，这是典型的 垂直格式（tidset / Eclat 思路）：用集合交代替事务扫描。

DFS 在视觉上像「穷举很多分支」，容易联想到排列；但分支数由「频繁项组合空间」决定，与阶乘级全排列不是同一量级的问题。

2. 性能会不会很差？最坏与典型

2.1 理论下界与最坏情况

挖掘「所有」频繁项集在一般情形下是 NP 难相关的问题：频繁项集数量本身在最坏情况下可以关于项数指数增长（经典例子：大量项几乎总在同一批文档共现）。因此 任何完整枚举算法在最坏情况下都不可能多项式有界。

2.2 本实现已有的工程护栏

  • minSupport：支持度不足的交集直接 continue，不再向下扩展（Apriori 单调性剪枝）。
  • maxItemsetSize：限制项集最大长度，组合深度有硬上限。
  • maxCandidateCount：候选项数量达上限即停止，保证最坏时延可控。
  • 先过滤非频繁 1-项集，缩小 DFS 的「字母表」。

2.3 何时会「仍然很重」

当频繁 1-项集很多、且它们在大片文档上高度共现时，即使剪枝，中间交集仍可能很多。此时瓶颈常在：BitSet 克隆与按位与的次数、以及 候选输出量。

3. 开源项目通常怎么做

思路 代表生态 特点
Apriori Weka、教学代码、早期库 逐层生成候选 + 剪枝；实现直观，大数据上常慢于 FP/Eclat。
FP-Growth Spark MLlib FPGrowth、Python mlxtend、许多工业管线 压缩 FP 树，避免显式候选爆炸；通常需额外工作才能拿到每项集的完整 tid 列表。
Eclat / 垂直 tidset SPMF、部分研究实现 与本仓库类似：交集求支持；tidset 稀疏时常配合压缩位图或 diffset。
LCM 系列等 频繁闭项集、最大项集专用算法 输出规模更小，适合「不需要全部频繁项集」的场景。

若你需要「项集 → 精确 docid 列表」且项集长度有界，垂直 tidset + 深度受限 DFS 在工程上很常见；大规模时会把 BitSet 换成更稀疏友好的结构并做并行。

4. 更好的优化手段（按投入从小到大）

  1. 项排序：按支持度升序排列项（FP-Growth / Eclat 常用），使早期分支更「稀疏」，剪枝更早。当前实现按 termId 数值序，未按频率重排。
  2. tidset 表示：稠密文档 ID 用 BitSet 往往很好；极稀疏或极大语料可评估 RoaringBitmap、有序 int 数组 + galloping 交集等。
  3. 减少克隆：复用缓冲区、原地与后恢复（若可证明安全）、或仅在叶子/输出时拷贝结果。
  4. 只挖闭项集或最大项集：若下游只需「代表性」频繁模式，输出量可下降一个数量级以上。
  5. 并行化：按首项或前缀分片，多线程独立子树（注意内存与上限协调）。
  6. 近似与采样：在线场景可对文档或项采样，用概率保证换时间。
没有「一种算法永远最快」：数据稀疏/稠密、最小支持度、是否只要闭集、是否要 tid 列表，都会决定最优实现。

5. 与本仓库设计文档的关系

仓库内 docs/posting-dedup-and-exclusive-fiset-design.html 已说明：垂直 term → tidset、交集扩展、以及 RoaringBitmap 等工程建议与本页一致。当前 Java 实现选择了 BitSet 与显式双上限，偏「实现简单、行为可预期」。

6. 小结

  • 不是全排列，是有剪枝的组合 DFS + 垂直 tidset 交集。
  • 最坏情况仍可指数级，这是问题本质；已通过 maxItemsetSize 与 maxCandidateCount 控制产品级风险。
  • 业界主流是 Apriori / FP-Growth / Eclat 及其变体；进一步优化多落在「项序、位图结构、少拷贝、闭/最大项集、并行」上。
