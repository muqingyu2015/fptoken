倒排共享存储与互斥频繁项集设计方案

目标：对“经常一起出现的词”进行压缩存储，特别是当多个词倒排表完全相同时，保证只存一份。

作者诉求：不仅要压缩，还要得到“互斥、最大长度、价值最高”的频繁项集合。

关键结论：先做“完全相同倒排表去重（可无损替换）”，再做“频繁项集共享核心倒排（增益型优化）”。

1. 需求拆解（按可落地顺序）

  1. 硬需求 A：若 P(a)=P(b)=P(c)=[1,2,3,4]，只存一份 [1,2,3,4]。
  2. 硬需求 B：能拿到“频繁项集对应的 docid 列表（TID set）”，不是只有 support。
  3. 硬需求 C：在候选频繁项集中选出“互斥 + 最大长度（或最大价值）”的集合。
只用标准 FP-Growth API 通常只能拿到 support，拿不到每个项集的原始 docid 列表，这是你遇到的核心缺口。

2. 先做无损去重：相同倒排表只存一份

2.1 数据结构

  • posting_blob(blob_id, compressed_doclist, size)
  • term_dict(term, blob_id, df, ...)

2.2 构建流程

  1. 每个词先生成规范化倒排（升序、去重、同编码）。
  2. 计算指纹（如 64-bit hash）作为候选 key。
  3. 同 hash 的倒排做字节级/元素级二次比较，确认完全相同后复用同一 blob_id。
  4. term_dict 里多个词可指向同一 blob_id。

这样就能精确满足你举例的“1234 只存一份”需求。

3. 频繁项集为什么仍然有价值

完全相同倒排只覆盖“等价词组”场景。更多时候是高度重叠而非完全相同。

例如：

  • P(a)=1,2,3,4,9
  • P(b)=1,2,3,4,10
  • P(c)=1,2,3,4,11

交集核心是 1,2,3,4。可存一份核心，再存每词残差，达到进一步压缩。

这时频繁项集对应的 docid 列表，实际上就是该项集的交集 TID set，可直接作为共享核心倒排。

4. 如何拿到“项集 -> docid 列表”

4.1 不用只看 FP-Growth 标准输出

建议使用“垂直数据格式”挖掘：term -> tidset，通过交集生成更大项集的 tidset。

  • 单项：T({a}) = P(a)
  • 扩展项集：T(X U Y) = T(X) ∩ T(Y)

这本质是 Eclat 思路，天然能产出“项集对应 docid 列表”。

4.2 工程实现建议

  • tidset 用 RoaringBitmap 存储，交集非常快。
  • minsup 过滤，避免候选爆炸。
  • 只保留长度 >= 2 且压缩收益为正的项集。

5. 互斥 + 最大长度：候选选择模型

5.1 候选定义

每个候选项集 S_i 携带：

  • 词集合：terms_i
  • 共享核心：core_i = intersect(postings of terms_i)
  • 权重：w_i（长度、支持度、节省字节数综合）

5.2 互斥约束

同一个词只能加入一个被选项集：对任意词 t，sum(x_i | t in S_i) <= 1。

5.3 目标函数

最大化 sum(w_i * x_i)，其中可设置长度优先：

w_i = alpha*|S_i| + beta*|core_i| + gamma*bytes_saved_i

这就是你说的“互斥最大长度频繁项集合”，严格可实现为加权集合打包。

6. 两级求解策略（你原思路的工程化版本）

层级方法用途
在线/日常 贪心：按 |S|、|core|、bytes_saved 排序后选不冲突集合 快，稳定，易维护
离线优化 OR-Tools CP-SAT 解 0-1 整数规划 追求更优上界，产出策略版本

推荐：贪心先给初解，CP-SAT 限时优化，超时则回退当前最优可行解。

7. 存储格式建议（保证可还原）

7.1 精确去重层（必须）

term 直接指向共享 blob，可无损还原。

7.2 频繁项共享层（可选增强）

  • 存 core_blob(S) 一份。
  • 每个词存残差：residual(t,S) = P(t) - core(S)。
  • 查询该词时：P(t) = core(S) U residual(t,S)（保证正确性）。
不要把项集交集直接替代词的完整倒排，否则会漏召回。必须配残差或保留原始倒排。

8. 落地里程碑（建议）

  1. M1：完成“相同倒排表去重”，验证空间收益和查询一致性。
  2. M2：实现 vertical tidset 挖掘，拿到“项集 -> docid 列表”。
  3. M3：实现互斥选择（先贪心），并输出收益报告。
  4. M4：接入 CP-SAT 离线求精，形成可切换策略版本。

9. 你当前需求的最终答案（简版）

  • “1234 只存一份”可以直接通过“倒排等价类去重”实现，且完全无损。
  • “项集对应 docid 列表”要用 tidset 方案（Eclat/垂直交集），不是只依赖标准 FP-Growth 输出。
  • “互斥最大长度”用加权集合打包建模，贪心 + OR-Tools 两级求解是最佳工程路径。
这套方案和你的原始思路完全一致，只是把“可落地的数据结构、约束模型、执行顺序”补齐了。
