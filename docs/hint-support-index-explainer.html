Hint 支持度回算：倒排索引优化说明

目标：解释为什么把 countRowsContainingAllTerms(currentRows, refs) 改成“倒排 + 最短 posting + 二分校验”后，性能会明显更好。

1. 原始做法（慢在哪里）

原始逻辑对每个 hint 都扫描整份 currentRows：

// 对每个 hint
for each aggregate in aggregated:
  support = 0
  for each row in currentRows:
    if row 包含 aggregate.refs 全部 term:
      support++

当 |aggregated| 和 |currentRows| 都大时，复杂度接近：

O(|aggregated| * |currentRows| * 匹配成本)

即使 aggregated 已经去重，row 侧仍被重复全表扫描，这是主瓶颈。

2. 优化做法（核心思想）

先一次性构建“只包含 hint 相关 term”的倒排索引：

  • termTable：term -> termId
  • postings[termId]：该 term 出现过的 rowIndex 有序数组

之后每个 hint 的支持度计算不再扫全表，而是：

  1. 找到 hint 中 posting 最短的 term 作为 anchor。
  2. 只遍历 anchor 的 rowIndex 列表。
  3. 对其余 term 的 posting 做 二分查找（Arrays.binarySearch）判断该 row 是否存在。

关键收益：从“每个 hint 扫所有 row”变成“每个 hint 只扫最短 posting”。

3. 你关心的点：contains 是不是暴力？

这里的 contains 不是 List.contains 线性查找，而是：

Arrays.binarySearch(rows, 0, size, rowIndex) >= 0

复杂度是 O(log N)，不是 O(N)。

4. 复杂度对比（近似）

方案 单个 hint 支持度回算 总复杂度特征
全表扫描 O(|rows| * hintLen * 行内匹配成本) O(|aggregated| * |rows| * ...)
倒排 + 最短 posting + 二分 O(minPostingLen * hintLen * log(posting)) 一次建索引 + 多候选复用

5. 一个小例子

假设 hint 是 [A, B, C]：

  • posting(A)=50w
  • posting(B)=2w
  • posting(C)=1.5w

优化后会选 C 当 anchor，只检查 1.5w 行，而不是扫全部行。

6. 为什么还要保留 aggregated

  • aggregated 负责减少 hint 维度重复（候选去重）。
  • 倒排索引负责减少 row 维度重复扫描。

两者是互补关系，不冲突。

7. 当前实现对应关系

  • buildHintSupportIndex(...)：建 termId 与 postings
  • countRowsContainingAllTerms(HintSupportIndex, refs)：按最短 posting 回算支持度
  • IntRowPosting.contains(rowIndex)：二分查找，不是线扫

8. 后续可继续优化（可选）

  • 把“anchor + 多次 binarySearch”升级为“posting 交集双指针”。
  • 按 requiredSupport 加早停（剩余最大可能支持度不足时提前退出）。
  • 对超高频 term 增加位图模式（按数据分布启用）。
