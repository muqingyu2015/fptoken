LXDB 集成说明（二）

主题：基于 LineFileProcessingResult.selectionResult 与 LineFileProcessingResult.derivedData 的双索引构建。

目标语义：高频词用于压缩索引，非高频词用于 skip index。

文件：docs/lxdb-integration-derived-data-and-selection-result.html · 日期：2026-04-22

1) 你当前的数据契约（已对齐）

  • selectionResult：互斥高价值词组（SelectedGroup）与统计信息，代表“主索引核心候选”。
  • derivedData.hotTerms：高频单词项及其 docId 列表，且已去除 selectionResult 中出现的词项。
  • derivedData.cutRes：原始行视图去除已选词项后的残余词项集合，可作为 skip index 的输入。

关键点：`buildDerivedData` 的语义是“先统计高频，再剔除 selection 中词项”，不是“先剔除再统计”。

2) LXDB 侧索引职责拆分

A. 高频压缩索引（主路径）

  • 输入：selectionResult.groups + derivedData.hotTerms
  • 用途：覆盖高频/高收益模式，提升压缩比与查询命中效率。
  • 策略：优先写入 SelectedGroup，再补充 hotTerms 的单项倒排。

B. Skip Index（补充路径）

  • 输入：derivedData.cutRes
  • 用途：承接非高频残余词项，降低漏召与极端 case 查询退化。
  • 策略：可按 docId 分块建立 skip pointers，支持快速跳跃。

3) 调用与落库流程（推荐）

rows -> ExclusiveFpRowsProcessingApi.processRows(...) -> LineFileProcessingResult -> [A] selectionResult + derivedData.hotTerms => 高频压缩索引 -> [B] derivedData.cutRes => skip index
LineFileProcessingResult out = ExclusiveFpRowsProcessingApi.processRows(rows, minSupport, minItemsetSize, hotThreshold); ExclusiveSelectionResult sel = out.getSelectionResult(); LineFileProcessingResult.DerivedData dd = out.getDerivedData(); // A: 主索引（高频） buildCompressedIndex(sel.getGroups(), dd.getHotTerms()); // B: 补充索引（残余） buildSkipIndex(dd.getCutRes());

4) 三条实现约束（避免后续踩坑）

  • 去重一致性：`hotTerms` 与 `cutRes` 已剔除 selected 词项，LXDB 二次处理不要重复剔除导致信息损失。
  • 阈值语义：`hotTermThresholdExclusive` 是严格大于（`count > threshold`），不是大于等于。
  • 分层稳定性：主索引只关心高频与收益，skip index 只关心补偿覆盖，不要混合排序规则。

5) 为什么这个拆法适合 LXDB

  • 高频路径与残余路径解耦，便于分别调参与独立优化。
  • 主索引压缩收益高，skip index 负责兜底，查询稳定性更好。
  • 接口边界清晰：`selectionResult` / `hotTerms` / `cutRes` 分别表达不同职责。

结论：你的设计是对的，且与当前实现语义完全一致，可直接作为 LXDB 索引分层方案落地。
