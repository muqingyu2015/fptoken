# fptoken 项目未来展望

> 本文档记录各角色对项目未来的长远规划思考。
> 重点是二进制检索场景下的膨胀率、性能和架构演进方向。
> 每个角色独立撰写，汇总在同一文件中。

## 1. Developer（开发工程师） — 未来规划

### 核心方向：从"挖掘"到"检索"的转变

当前项目聚焦于频繁项集挖掘，但最终目的是**二进制检索压缩**。未来应逐步从挖掘算法优化转向检索场景的端到端优化。

### 1.1 膨胀率控制（P0）

**问题：** 频繁项集的最终目的是压缩倒排索引，减少查询膨胀。但当前没有膨胀率指标。

**未来要做的事：**
1. **引入膨胀率监控指标：** 每次挖掘后输出压缩前后的索引大小对比
2. **膨胀率驱动的参数调优：** 根据膨胀率自动调整 minSupport、maxItemsetSize 等参数
3. **组合有效性评估：** 不是所有频繁组合都值得编码——只保留能显著降低膨胀率的组合

**预期效果：** 用户可以直接看到"这个配置节省了 X% 的索引空间"。

### 1.2 检索性能评估（P1）

**问题：** 频繁项集压缩后，检索性能如何？没有基准数据。

**未来要做的事：**
1. **构建检索基准测试：** 对比压缩前后 100/1000/10000 次查询的平均延迟
2. **压缩率 vs 查询延迟 trade-off 分析：** 找出最佳平衡点
3. **动态编码策略：** 根据查询负载模式调整压缩策略

**预期效果：** 明确告诉用户"用这个配置，索引大小减少 X%，查询变慢 Y%"。

### 1.3 自动化参数调优（P1）

**问题：** 当前参数（minSupport、采样比率等）需要人工调整。

**未来要做的事：**
1. **自动探索参数空间：** 跑几个小规模样本，找到最优参数组合
2. **自适应采样比率：** 根据数据分布自动调整
3. **分阶段调优：** 先粗调（快速缩小范围），再精调（在小范围找最优）

**预期效果：** 用户只需要提供数据和目标（"省内存"或"快查询"），系统自动找到最佳配置。

### 1.4 增量更新支持（P2）

**问题：** 当前每次挖掘都需要全量重建索引。

**未来要做的事：**
1. **增量索引构建：** 新文档加入时只更新增量部分
2. **增量频繁项挖掘：** 支持增量更新频繁项集
3. **定时合并：** 定期合并增量到全量索引

**预期效果：** 每天新增 10% 数据时，增量更新只需全量重建的 10% 时间。

### 1.5 端到端 Pipeline（P2）

**未来架构：**
```
输入文档 → 分词 → 索引构建 → 频繁项挖掘 → 组合编码 → 索引压缩 → 检索服务
                           ↑                                  ↓
                     采样（可选）                       膨胀率反馈
```

每个步骤的输入输出都是标准化的，可以独立替换和优化。

## 2. Architect（架构师） — 架构演进规划

### 2.1 当前架构的局限

```
当前：全量文档 → 索引构建（98%时间）→ 频繁项挖掘 → 互斥挑选 → 输出
```

核心问题：**索引构建和挖掘耦合在一起**，无法独立扩展。

### 2.2 未来架构演进路径

#### 阶段一：解耦（短期 1-3 个月）

```
                 → 索引构建服务（可独立部署）
全量文档 → 分词 → → 频繁项挖掘服务（可独立部署）
                 → 采样引擎（可选，可独立部署）
```

**好处：**
- 各组件可独立扩展
- 索引构建可用多机并行
- 挖掘服务可以根据数据量调整资源

#### 阶段二：流式处理（中期 3-6 个月）

```
持续文档流 → 实时分词 → 增量索引 → 定时挖掘 → 编码压缩 → 检索集群
                            ↑                     ↓
                      采样质量监控 ←── 膨胀率反馈 ──
```

**好处：**
- 支持实时/近实时索引更新
- 自动质量监控和回滚
- 膨胀率驱动的自适应调优

#### 阶段三：自优化系统（长期 6-12 个月）

```
文档流 → [自动调优引擎]
               ↓
          [参数探索器] ←── 采样质量评估
               ↓
          [挖掘引擎] → [编码压缩] → [检索集群]
                             ↓
                        [膨胀率/延迟监控]
                             ↓
                        [自动调整配置]
```

**核心思路：** 系统自动探索最佳配置，持续优化膨胀率和查询延迟。

### 2.3 关键技术选型

| 组件 | 当前 | 未来 |
|---|---|---|
| 索引构建 | 单机 HashMap + BitSet | 分片并行 + 可选 RoaringBitmap |
| 挖掘引擎 | Beam Search（单线程） | 可并行 Beam Search |
| 编码压缩 | 无（输出组合列表） | 自动编码器 |
| 存储 | 无 | 压缩倒排索引 |
| 检索 | 无 | 二进制检索引擎 |

### 2.4 膨胀率控制架构

**核心指标：**
- **原始索引大小：** 不压缩的倒排索引大小
- **压缩后索引大小：** 应用频繁项编码后的索引大小
- **膨胀率：** 压缩后大小 / 原始大小（越低越好）
- **查询延迟：** 压缩前后平均查询延迟

**控制策略：**
1. 挖掘阶段：只保留能降低膨胀率的组合
2. 编码阶段：选择最紧凑的编码方案
3. 检索阶段：平衡压缩率和查询速度

## 3. Java Perf Guru（Java 性能专家） — 性能演进规划

### 3.1 当前性能基线

| 场景 | 100K docs | 1M docs（预估） |
|---|---|---|
| 索引构建 | ~2.5s | ~25s |
| 频繁项挖掘 | ~30ms | ~300ms |
| 互斥挑选 | ~5ms | ~50ms |
| 总计 | ~2.5s | ~25s |

### 3.2 未来性能目标

| 场景 | 当前 | 目标（1年内） | 关键优化 |
|---|---|---|---|
| 100K docs | 2.5s | <1s | 并行索引、自定义哈希表 |
| 1M docs | 25s（预估） | <5s | 分片并行、流式处理 |
| 10M docs | 250s（预估） | <30s | 分布式索引、增量更新 |

### 3.3 性能优化路线图

**短期（1-3 个月）：**
1. **自定义 int[] 哈希表（-30% 首次扫描时间）**
   - 替代 HashMap 统计频率
   - 只存储 hash + count，通过 minSupport 后存 byte[]
2. **预分配 HashMap 容量（-5% rehash 开销）**
3. **JVM 参数调优（-10% GC 停顿）**

**中期（3-6 个月）：**
1. **分片并行索引构建（2-3x 加速，4 核）**
2. **并行 Beam Search（1.5-2x 加速）**
3. **增量索引更新（新增文档的 O(1) 更新）**

**长期（6-12 个月）：**
1. **索引构建与检索的联合优化**
2. **基于查询负载的自适应编码**
3. **硬件感知优化（SIMD、NUMA）**

### 3.4 膨胀率与性能的权衡

**核心洞察：**
- 压缩率越高 → 索引越小 → 内存越省 → I/O 越少
- 但解码开销越大 → 查询越慢

**最佳平衡策略：**
1. 热门组合用快速编码（解码快）
2. 冷门组合用紧凑编码（省空间）
3. 根据查询频率动态调整编码策略

## 5. Search Guru（搜索专家） — 检索场景规划

### 5.1 二进制检索与频繁项集的关系

**核心逻辑：**
```
原始倒排索引：
  term1 → [doc1, doc5, doc8, ...]
  term2 → [doc2, doc4, doc7, ...]

频繁组合编码后：
  组合 {term1, term2, term3} → 编码为短前缀 P1
  P1 → [doc1, doc5, ...]  (只存储一次)
  
  查询 "term1 AND term2" 时：
  原始：取 term1 和 term2 的 doclist 做交集
  编码后：直接取 P1 的 doclist (更快！)
```

**关键指标：**
- **膨胀率：** 编码后索引大小 / 原始索引大小
- **查询加速比：** 原始查询延迟 / 编码后查询延迟
- **组合覆盖率：** 被编码的查询比例

### 5.2 未来改进方向

#### 方向一：查询驱动的组合挖掘（P0）

**问题：** 当前挖掘所有频繁组合，但很多组合在实际查询中用不到。

**方案：** 根据查询日志分析，优先挖掘**高频查询模式**的组合。

**效果：** 同样的编码空间，覆盖更多实际查询，膨胀率降低更明显。

#### 方向二：多级编码策略（P1）

```
一级编码：超高频组合 → 最短前缀（解码最快）
二级编码：中频组合 → 中长度前缀
三级编码：低频组合 → 不编码（原始存储）
```

**效果：** 高频查询加速最大，低频查询不受影响。

#### 方向三：动态编码调整（P2）

- 根据查询负载变化动态调整编码策略
- 热门组合上升时自动升级编码
- 冷门组合下降时自动降级

### 5.3 膨胀率优化策略

| 策略 | 膨胀率降低 | 查询性能影响 | 实现复杂度 |
|---|---|---|---|
| 只编码 Top-10 组合 | 10-20% | +5% | 低 |
| 多级编码 | 30-50% | +10% | 中 |
| 查询驱动挖掘 | 40-60% | 无影响 | 中 |
| 自适应编码 | 50-70% | +5-15% | 高 |

### 5.4 推荐路线

1. **先做膨胀率监控**（知道当前状态）
2. **再实现查询驱动挖掘**（最有效）
3. **再实现多级编码**（平衡膨胀率和性能）
4. **最后做自适应调整**（长期演进）

## 6. Big Data Guru（大数据专家） — 大规模场景规划

### 6.1 规模挑战

| 文档数 | 当前方案 | 问题 |
|---|---|---|
| 100K | ✅ 2.5s | 完美 |
| 1M | ⚠️ ~25s | 可接受，但慢 |
| 10M | ❌ ~250s | 不可接受 |
| 100M | ❌ ~2500s | 需要根本性改变 |

### 6.2 分片并行架构

```
文档流
  ├── 分片1 (10K docs) → 独立索引构建 → 局部频率统计
  ├── 分片2 (10K docs) → 独立索引构建 → 局部频率统计
  ├── 分片3 (10K docs) → 独立索引构建 → 局部频率统计
  └── ...
         ↓
    合并节点：汇总频率 → 分配全局 termId → 合并 BitSet
```

**关键设计：**
- 分片大小根据内存和 CPU 核心数自适应
- 合并操作是 O(唯一词数) 的轻量操作
- 支持水平扩展（增加分片 = 线性加速）

### 6.3 流式处理架构

```
持续文档流 → 窗口缓冲（如 10K docs/batch）
                 ↓
          批处理索引构建
                 ↓
          增量合并到全量索引
                 ↓
          定期全量重建（如每天一次）
```

**关键设计：**
- 新文档延迟 < 1 分钟可见
- 增量合并开销 < 全量重建的 10%
- 支持 Exactly-Once 语义

### 6.4 混合架构

```
实时路径：新文档 → 增量索引 → 实时检索
批处理路径：全量数据 → 全量索引 → 频繁项挖掘 → 编码压缩 → 替换实时索引
```

**好处：**
- 新数据秒级可见
- 压缩更新每天一次
- 用户无感知切换

## 7. OS Tuning Guru（操作系统调优专家） — 系统级规划

### 7.1 当前系统瓶颈

| 资源 | 当前消耗 | 未来挑战 |
|---|---|---|
| CPU | 单核 100%（索引构建） | 多核并行后需要 CPU 亲和性 |
| 内存 | 1-5GB（100K-1M docs） | 10M+ docs 需要数十 GB |
| I/O | 无（全内存） | 流式处理后需要高效磁盘 I/O |
| 网络 | 无（单机） | 分布式后需要低延迟网络 |

### 7.2 系统优化路线

#### 短期：JVM 调优
```bash
-Xms4g -Xmx4g          # 固定堆
-XX:+UseG1GC           # G1GC
-XX:MaxGCPauseMillis=50
-XX:ParallelGCThreads=4
```

#### 中期：容器化部署
```yaml
# Docker/K8s 资源限制
resources:
  limits:
    cpu: 4
    memory: 8Gi
  requests:
    cpu: 2
    memory: 4Gi
```

#### 长期：硬件感知优化
- NUMA 感知的内存分配
- SIMD 加速的 BitSet 操作
- 大页内存（Huge Pages）减少 TLB miss

### 7.3 监控体系

**需要监控的指标：**
- CPU 使用率（用户态/内核态/GC）
- 内存使用（堆/非堆/直接内存）
- GC 频率和停顿时间
- 磁盘 I/O（流式处理后）
- 网络延迟（分布式后）
