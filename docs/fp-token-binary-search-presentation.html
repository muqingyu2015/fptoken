LXDB 能力介绍

二进制检索

在海量文件、日志、报文等二进制内容里，快速、准确地按片段查找「谁含有某段内容」——像搜关键词一样简单。

1 / 9
01 · 是什么

什么是二进制检索？谁需要？

「这段字节 / 字符，有没有出现在某条记录里？」——不能靠分词，只能按字节片段匹配。

🔒 安全合规

日志、样本库中搜特征码、恶意片段

📁 文件样本

按片段找相似文件、溯源归类

📡 协议设备

报文、固件中定位异常或目标

共同需求：海量数据、秒级响应、不能离线全盘扫。

2 / 9
02 · 业界主流

主流产品怎么做？问题在哪？

Elasticsearch · 分词 + 小片段建索引→ 膨胀率高，存储写入贵
OpenSearch · 同 ES 路线→ 膨胀、运维与同步成本高
Apache Solr · 分词检索→ 偏文本，二进制需转格式
ClickHouse · Skip 跳跃 / 概率索引→ 高命中片段性能极差
PostgreSQL · LIKE / 三元组索引→ 海量慢，二进制非原生
MySQL · 全文索引→ 二进制片段支持弱
Oracle · 全文检索组件→ 许可贵，非二进制优先
MongoDB · 文本索引 / 正则→ 大数据量正则很慢
Splunk · 日志专用索引→ 专有、费用高、与业务库割裂
HBase 等 · 离线扫 / 外挂检索→ 实时弱、架构复杂

共性：几乎没有「为二进制片段检索而生」的产品——难同时做到快、准、省、与数据库一体。

3 / 9
03 · 行业痛点

这些方案普遍面临什么问题？

❌ 查得慢

海量数据下全表扫，秒级变分钟级

❌ 存得贵

全片段索引导致存储与写入成本飙升

❌ 写得太重

入库预处理拖慢实时业务

❌ 与库脱节

外挂检索，SQL / 权限 / 运维难统一

核心矛盾：既要像搜文字一样快，又要按字节片段准，还要数据库一体交付。

4 / 9
04 · LXDB 方案

我们是怎么解决的？

入库→ 内容指纹→ 分级存储→ 按需查询→ 秒级命中

入库时算清楚

常出现片段重点保留，索引不无限膨胀

查询时读得少

只加载本次相关索引，IO 可控

热点优先

高频片段单独加速，减少无效比对

多段联合确认

长查询拆段、全部命中才算，降低误报

用户仍用 SQL / 表查询，能力内置于 LXDB，无需另购检索组件。
