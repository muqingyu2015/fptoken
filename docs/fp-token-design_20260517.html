1. 代码解读

1.1 模块在做什么

FP Token 为 LXDB/Lucene 提供按二进制滑窗切出的子串作为词项的倒排能力： 索引阶段对字段字节流做 n-gram 式滑窗，写出带固定 12 字节头部的词项； 在段合并写 BlockTree 时，按组号（index_id + group_id，共 6 字节）聚合多个 term， 在组内做热词挖掘、ngram 位图索引、块级闭包判定，再落盘到倒排与侧车 bit 文件。

1.2 核心类职责（按调用链）

类包职责摘要
FPToken / FpTokenAnalyzer token 分析阶段：二进制滑窗分词，输出带 FpTokenTermLayout 头部的 BytesTermAttribute。
FPBlockTreeTermsWriter api 写段入口：遍历 Terms#iterator_fp()，驱动 FpTokenBlockOrchestrator。
FpTokenBlockOrchestrator api 块编排：按词项级别分流「高候选原样组」与「可合并重建组」，换组/闭块时 flush。
FpGroupDataOriginal dataset.block 高级别路径：ingest 后若体量达标，借助已有 fpBits 位图原样写出（不重算 ngram 热词）。
FpGroupDataRebuild dataset.block 可合并路径：缓冲组内 hot/common 词表 → FpGroupHotNgramRebuild → FpGroupHotNgramBitIndex → writefp + bit 持久化。
FpGroupHotNgramRebuild dataset.block 从 common 载荷统计 byte ngram(1~8) 频率，超过阈值升格为 hot，并把 common 的 doc 合并进对应 hot ngram。
FpGroupHotNgramBitIndex dataset.block 构建/读写 8×256 的 FixedBitSet 矩阵（hot 与 common 各一套），支持全量读与按需读。
FPDocList dataset.common 单 term 的 doc 列表：递增 int[] 或 SparseFixedBitSet，供 writefp 遍历。
FpBlockInfo dataset.common 每组 flush 后在 meta 中记录 bit 区偏移、步长、维度；查询侧 fpBits 定位数据。
FpTokenTermLayout dataset.common 12 字节 FP 头：index_id、group_id、level、hot 标记、termIndex+删除位 + 载荷。
FpFilteredTermsEnum api 多子索引合并时，在词项前 2 字节写入分片 index，保证字节序可区分来源。

1.3 词项头部布局（12 字节）

[index_id:2][group_id:4][group_level:1][hotmark:1][termindex+isDelTerm:4][payload...]
  • index_id：短整型 sortable，合并时可被替换为逻辑分片号。
  • group_id：组内写出时由 FpTokenBlockOrchestrator.groupIndex 递增分配。
  • group_level：块级别，与 FpTokenBlockLevelPolicy 配合。
  • hotmark：1=热词，0=普通词（写入时由重建结果决定）。
  • termindex+isDelTerm：高 31 位为 term 序号，最低位为删除占位标记。

1.4 编排器关键分支（FpTokenBlockOrchestrator）

  • acceptTerm：若 readLevel(term) >= targetLevel → 进入 FpGroupKVOriginal；否则进入 FpGroupKVRebuild。换组时 flush 高组；可合并组换组时若已达闭块条件则 flush common。
  • flushHighGroup：用 shouldCompleteBlock(1, targetLevel, docs, terms) 判断。达标 → terms.fpBits(index_id, group_id, null, null) 取已有位图，FpGroupDataOriginal#flushto；不达标 → mergeInto 可合并组。
  • flushCommonGroup：FpGroupDataRebuild#flushto(orchestrator, false)，完整走热词重建 + 位图构建 + 写出。

1.5 热词重建与位图（精读）

FpGroupHotNgramRebuild.execute 三步：

  1. countNewHotNgramsFromCommon：对每个 common 词项载荷做 1~8 字节滑动 ngram，词项内去重；已存在于 hot 表的切片跳过；累加频率。
  2. buildFinalHotTerms：频率 ≥ 阈值（如 32）的 ngram 加入最终 hot 表（保留旧 hot）。
  3. mergeCommonDocsIntoFinalHot：按较长 ngram 优先，把 common 的 FPDocList 合并到匹配的热词 ngram 上（子域去重避免重复合并）。

FpGroupHotNgramBitIndex.execute：

  • 热词侧：按 hotTermToOrder 编号，对载荷每个 ngram 在对应桶的 FixedBitSet 上 set(order-1)。
  • 普通词侧：独立 8×256 矩阵；若 ngram 切片已是 hot 键则不写 common 侧。
  • 桶规则：长度 1 → 字节值作桶号；长度 2~8 → 多项式哈希折叠到 0..255。

2. 基本原理

2.1 二进制指纹检索

传统分词依赖词典；FP Token 将字段视为原始字节序列，通过固定步长的 64bit 滑窗（见 BinarySlidingWindowApi） 产生大量重叠子串。相似二进制内容在倒排上共享子串，从而支持模糊/部分匹配类检索（具体查询算子由上层 Lucene 查询实现）。

2.2 组（Block Group）与级别（Level）

同一 index_id + group_id 下的 term 在写段时被当作一个逻辑块： 共享组级 doc 并集 distinctDocUnion，并共享一块 ngram 位图索引（FpBlockInfo 指向的数据）。 词项自带的 group_level 表示该 term 在索引 pipeline 中已满足的闭块规模； 写段编排器另根据段规模计算 targetLevel，决定哪些 term 必须参与「合并重建」。

2.3 热词 vs 普通词

  • 热词：高频或显式标记的 ngram，倒排 posting 更短、查询可先过滤热词位图。
  • 普通词：组内其余载荷；位图索引排除与热词重复的 ngram，减少冗余。

2.4 Doc 列表的自适应存储

FPDocList 假设单 term 在单段内 doc 多为递增追加； 一旦出现重复、乱序或超过 1024 条，立即升级为 SparseFixedBitSet。 组合并时若双方均为 sparse，使用 SparseFixedBitSet#or(BitSetIterator) 快速归并。

2.5 闭块策略（统计学意义）

FpTokenBlockLevelPolicy.shouldCompleteBlock(rate, targetLevel, distinctDocs, distinctTerms) 当组内去重文档数或不同词项数达到级别阈值（低级别约 1000，高级别约 100000，可乘以 rate）时认为块「成熟」，可以写出。 高级别 term 若统计未达标则降级并入可合并组，避免大量小碎块。

3. 设计思路

3.1 双路径：透传 vs 重建

设计动机：已达目标级别且统计充足的组，在分析阶段已生成稳定的热词与位图； 写段时若再跑 ngram 重建会浪费 CPU。FpGroupDataOriginal + terms.fpBits(...) 实现读已有位图、原样 writefp。 未达标或低级别 term 进入 FpGroupDataRebuild，在内存中聚合后统一重建，保证热词表与位图一致。

3.2 热词挖掘与位图分离

  • 重建阶段用频率阈值从 common 升格 hot，并回写 doc 列表，使倒排与位图语义一致。
  • 位图上 hot/common 两套独立的 8×256 FixedBitSet，位宽各自为 hotNumBits / commonNumBits，避免编号空间耦合。
  • 查询侧可按 ngram 长度与桶号 O(1) 定位候选 term 集合（位图 AND/OR 由上层实现）。

3.3 磁盘局部性：交错写入

FpGroupHotNgramBitIndex#flushto 按 hot[li][b], common[li][b] 交错写入 bit 文件； FpBlockInfo 只存 fpBanksHot00、fpBanksCommon00 与单格字节步长， 任意 (li,b) 偏移可算术推导。利于顺序读与按需 seek 单桶。

3.4 键设计：FpTermKey

TreeMap<FpTermKey, …> 使用预计算 hash 的字节键； ngram 扫描中的临时查找用 viewOf（零拷贝），持久 map 用 copyOf，平衡 CPU 与内存。

3.5 多索引合并

FpFilteredTermsEnum 在词项最前写入 sortable index_id， 与 FpTokenTermLayout 头部协同，使 MultiTerms 合并后仍可按分片与组定位 fpBits。

5.1 查询侧（概念）

通过 FieldReader#fpBits 按 index_id、group_id 加载位图； 全量：readfrom(in, blkinfo)； 按需：readfromBanksSelective(in, blkinfo, loadHot, loadCommon)，未选中的格为 null。

6. 持久化格式

6.1 FpBlockInfo（写入字段 meta，经 BlockTree finish）

字段类型说明
FORMAT_VERSIONint当前为 1
fpBanksHot00longbanksHot[0][0] 在 bit 文件中的偏移
fpBanksCommon00longbanksCommon[0][0] 偏移
bytesPerHotSerializedint单个热词 FixedBitSet 序列化长度
bytesPerCommonSerializedint单个普通词 FixedBitSet 序列化长度
hotNumBits / commonNumBitsint位图逻辑长度
hotCount / commonCountintterm 数量统计

6.2 Bit 区布局（termsbit 文件）

对每个 (ngramLen=1..8, bucket=0..255) 按序写入：
  FixedBitSet_hot[li][b]   // DataOutput.writeBits
  FixedBitSet_common[li][b]

偏移计算（pair = li * 256 + b）：
  hotOffset   = fpBanksHot00   + pair * (bytesPerHot + bytesPerCommon)
  commonOffset = hotOffset + bytesPerHot

序列化格式与 Lucene FixedBitSet#writeBits / IndexInput#readBits(maxDoc) 一致。

7. 与 Lucene 的集成点

  • 字段识别：Lucene80FPSearchConfig.isFpField(field) — 后缀 _bfp（二进制）或 _sfp（字符串）。

9. 运维与扩展建议

9.1 调参入口

  • FpTokenBlockLevelPolicy.resolveTargetBlockLevel — 段规模 → 目标级别。
  • FpGroupHotNgramRebuild.execute(..., freqThreshold) — 热词频率阈值（当前调用处为 32）。
  • shouldCompleteBlock(rate, ...) — 闭块判定比例（common 换组 flush 使用 rate=3）。
  • FPDocList.MAX_ARRAY_DOCS — 数组模式上限（1024）。

9.2 性能关注点

  • 重建路径中 countNewHotNgramsFromCommon 对每个 common term 做 O(载荷长度×8) 扫描，是大头 CPU。
  • 位图构建为 8×256×2 次 set 操作，热词/普通词分表控制位宽。
  • 按需读位图可减少 IO，适合查询只触及少数 ngram 长度与桶的场景。

9.3 测试建议

  • 单组小 doc 数：验证 FPDocList 数组路径与乱序升格 sparse。
  • 高低级别 term 混合：验证降级 merge 与 original 透传两条路径。
  • round-trip：flushto → readfrom / readfromBanksSelective 位图一致性。
  • 多子索引：FpFilteredTermsEnum 前缀与 fpBits(index_id, …) 对齐。

9.4 版本兼容

FpBlockInfo.FORMAT_VERSION = 1；修改 on-disk 布局时必须递增版本并在 readfrom 分支处理。
