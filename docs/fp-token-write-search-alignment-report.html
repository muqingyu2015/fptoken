4. 透传：旧组读图、新组落盘（设计，非 bug）

// FpTokenBlockOrchestrator.flushHighGroup — 当前实现
final int logical_group = FpTokenTermLayout.readGroupIdFromIndexAndGroupKey(group_original.key);
FpGroupHotNgramBitIndex bits = terms.fpBits(index_id, logical_group, null, null);

final int new_group_id = groupIndex.incrementAndGet();
group_original.val.flushto(this, bits, new_group_id);
  • logical_group（6 字节流式键）：只在读索引已有位图时用。
  • new_group_id：本段倒排、fpblock_list、查询 fpBits 统一用新号。
  • FpGroupDataOriginal.flushto 内不再 incrementAndGet()，避免双次递增。

6. 剩余关注（非代码 bug）

透传集成风险（FpTokenBlockOrchestrator#flushHighGroup）
  • 本段首次出现某逻辑组时，fpBits(logical_group) 可能无历史位图。
  • 透传 posting 已变但复用旧图 → 位图 termIndex 与倒排不一致。
  • 已降级 mergeIntoRebuild 的组不应再走透传读图。

建议：LXDB 写段后对新 fpblock_list key 做 fpBits round-trip + 抽样 termIndex seek。

可选语义澄清（已实现为 seek 成功才算 hotHit）
若热词位图有置位但所有 seek 均失败，当前仍会扫 common。与「del 命中即 hotHit、不扫 common」不冲突； 若产品要求「位图非空即不扫 common」，需改 searchSliceInGroup 判断条件（当前未要求）。
