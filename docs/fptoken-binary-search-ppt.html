02 / 07

什么是「二进制检索」？

在不依赖分词器的前提下，把文档正文切成固定窗宽的字节指纹（ngram 载荷）， 写入 Lucene 倒排；查询时用字节切片 + 位图桶在索引里做子串式命中，而不是对全文做线性扫描。

  • 检索对象：原始二进制 / UTF-8 字节流，不是「词」。
  • 典型场景：日志、代码、JSON 宽表、海量稀疏列（每列单独列名前缀）。
  • 要解决的问题：
    • 大文本里「包含某段字节模式」的模糊 / 子串检索；
    • 在可接受存储下仍保持可 seek 的索引结构（BlockTree + FP 扩展）；
    • 避免纯暴力扫描与纯 NGRAM 全量膨胀两种极端。
词项布局：FpTokenTermLayout = 列名 + 14B 头 + ngram 载荷

本项目的技术特点

能力做法收益
频繁项集减存储 FpGroupHotNgramRebuild：在组内统计 common 的 ngram 共现，达阈值升格为热词；长模式向短锚点合并 doc，并受 hotDownTierBudget 约束 少写重复 posting，控制膨胀
按块跳跃 FpGroupHotNgramBitIndex：6 层 × 256 桶双套位图（hot / common）；查询先桶定位再 seek，可向下探若干层子串 跳过大量无关 term，降低 IO
闭块分级 FpTokenBlockLevelPolicy：按组内 distinct doc / term 数决定级别；大组透传，小组重建 merge 时大组免重算
JSON 稀疏列 term 前缀带可变长列名；换列即刷组，查询只 seek 目标列 百万级 key 不混桶
互斥项集选型 ExclusiveFrequentItemsetSelector：挖掘后再做互斥打包，避免模式重叠冗余 候选模式可执行、可比较
写段入口：FPBlockTreeTermsWriter · 编排：FpTokenBlockOrchestrator
04 / 07

对比主流方案 ①：NGRAM 暴力分词

原方案劣势

  • 膨胀率极高：窗宽 64、步长 32 时，长文档产生海量几乎互不重复的 term。
  • 倒排与 FST 体积随文档长度近线性暴涨，merge 易 OOM / 序错乱。
  • 高频字节片段与低频片段同等对待，存储无法「向结构要效率」。

我们的优势

  • 组内先挖频繁 ngram，升格热词；common 在预算内向热词合并 doc。
  • 位图标记「哪些 termIndex 在该桶出现」，查询先桶后 seek。
  • 写出前按最终 term 字节字典序排序，满足 BlockTree 约束。

一句话：不是「每个窗都建索引」，而是「先统计、再升格、再跳跃」。

05 / 07

对比主流方案 ②：Skip Index

原方案劣势

  • 高频词问题：常见字节片段出现在极多 doc，skip 链在该 term 上几乎失效。
  • Posting 极长，解码与合并成本集中在少数「超级 term」。
  • 对二进制子串场景，难以区分列 / 组，易跨列误命中。

我们的优势

  • 热词 / 普通词分流：热词承载主要 mass，common 侧位图可跳过已是热词的切片。
  • hotDownTierBudget 控制向下拼层数，平衡召回 vs 误命中。
  • 列名前缀 + FpSearch 校验，查询只在目标 JSON 列上比对 payload。
06 / 07

对比主流方案 ③：暴力匹配 & 全量重建

原方案劣势

  • 每次 merge 对整组重算热词 + 重区位图，CPU 与内存峰值高。
  • 已达闭块条件的大组仍走重建路径，浪费上一段已炼好的结构。
  • 热/普分批写出易导致 term 字典序错乱（已通过排序写出修复）。

FpGroupDataOriginal 的优势

  • 组内 doc 数 / term 数达标时：透传写出，复用合并前段的 fpBits 位图。
  • 热词与普通词在组内缓冲，落盘时按最终 term 排序统一 writefp。
  • 未达标才 mergeIntoRebuild 降级，由 FpGroupDataRebuild 重新挖掘——能透传就不重炼。

编排逻辑：FpTokenBlockOrchestrator.flushHighGroup → 达标则 Original.flushto，否则并入 Rebuild。
