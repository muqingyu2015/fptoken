buildWithSupportBounds 性能优化复盘

针对性能报告中 “TermTidsetIndex.buildWithSupportBounds 占热路径约 72%” 的专项优化说明。

项目：fptoken · 文档日期：2026-04-24

1. 问题背景

  • 热点审计显示：`buildWithSupportBounds` 是主瓶颈，明显高于 miner/picker。
  • 该方法本身包含三段：first pass（频次扫描）/ filter prepare（筛选映射）/ second pass（位图设位）。
  • 在词项规模上来后，排序与内层访问开销会被放大。

2. 本次已落地改进

优化点 改动前 改动后 收益预期
支持度排序算法 按 `(support, termId)` 的 插入排序 O(n²) 构造 `long` 复合键 + `Arrays.sort(long[])`（O(n log n)） 词项数上来时明显降耗
second pass 内层访问 循环中频繁 `List.get()` 与多次间接寻址 提前缓存 `originTermIdToFiltered` 和 `BitSet[]` 数组 减少热点分支与边界检查开销

3. 关键代码思路（简化）

3.1 排序由 O(n²) 改为 O(n log n)

long key = (((long) support) << 32) | (termId & 0xffffffffL); keys[termId] = key; Arrays.sort(keys);

3.2 位图设位内层使用数组直达

int[] map = filteredStage.originTermIdToFiltered; BitSet[] tidsets = filteredStage.tidsetsByTermId.toArray(new BitSet[filteredSize]); ... int filteredId = map[originTermId]; if (filteredId >= 0) { tidsets[filteredId].set(docId); }

4. 验证与稳定性

  • 变更后已执行项目回归测试：通过（0 failed）。
  • 功能语义保持不变：仍按支持度边界过滤 + 构建 term->doc 位图。
  • 风险控制：本次改动不涉及协议变更、不改业务输出结构。

5. 结果解读（为什么报告里 first pass 仍高）

  • 报告显示 first pass 依旧占比较高，这是符合预期的：它承担了最重的词项扫描与哈希映射工作。
  • 这次优化主要先消除了可确定的次优实现（O(n²) 排序与内层间接访问）。
  • 下一阶段若继续压缩 first pass，需要更深层方案（如批次局部去重结构、哈希布局优化、流水化扫描）。

6. 后续可继续做的 P1/P2

  • P1：`buildDerivedData` 减少重复遍历（目前在报告中也占高比重）。
  • P1：first pass 的哈希路径做批次去重与局部缓存，降低重复 hash 开销。
  • P2：按数据集特征分层参数模板，减少极端场景的无效提示词放大效应。

建议节奏：先“热点可确定优化”再“结构性重构”，避免一次性大改带来风险。
