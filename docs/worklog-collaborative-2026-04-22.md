## 1. Developer（开发工程师） — 工作日志

### 任务目标
对 fptoken 项目进行架构优化和性能优化，重点解决大词表场景下的索引构建瓶颈（占 97-99%）和采样质量下降问题。

### 思考过程

#### 阶段一：瓶颈定位

**思考：** 项目性能慢在哪里？是算法问题还是实现问题？

**做法：** 编写 `BottleneckProfiler.java` 对整个 pipeline 进行精确计时。

**发现：** 索引构建占 **98.2%**，频繁项挖掘只占 **1.1%**，互斥挑选占 **0.5%**。

**结论：** 所有优化资源应集中在索引构建上，采样对挖掘阶段的优化（占 1.1%）无法绕过索引构建瓶颈。

#### 阶段二：采样策略重构

**思考：** 老板要求"采样一定要做好"，但采样只优化挖掘阶段，索引构建才是瓶颈。怎么办？

**遇到的问题：**
1. 原始采样策略先建采样集索引，再回算全量索引，建两次索引导致成本翻倍
2. 大词表下采样结果严重下降（doc-refs 差异达 -19254）

**尝试的方案：**
- **方案 A：** 在采样集上单独建索引（两次建索引 → 成本翻倍 ❌）
- **方案 B：** 先建全量索引，从全量 tidsets 位过滤采样（只建一次索引 ✅）
- **方案 C：** 先建全量索引，采样集上单独建索引，回算时做 termId 映射（复杂且成本高 ❌）

**最终方案：** 方案 B + minSupport 缩放
- 全量建索引一次 → 随机选择采样文档 → BitSet 位过滤 → 采样挖掘（minSupport 按比例缩放）→ 全量回算
- 采样比例从 20% 提升到 30%

**质量验证：** 通过 `SamplingCompareTest`、`LargeScaleTest`、`LargeVocabTest` 三个测试验证，最终 Groups diff=0，doc-refs diff=0。

#### 阶段三：两阶段索引构建

**思考：** 索引构建占 98%，如何优化？低频词占唯一词 75-85% 但几乎不贡献结果。

**做法：** 实现 `buildWithSupportBounds`，先统计频率（HashMap），只对高频词分配 BitSet。

**遇到的新问题：** 两阶段需要两次文档扫描，第二次扫描每个词都要 `new ByteArrayKey(rawTerm)` 做 HashMap 查找，产生大量数组拷贝。

**解决：** 创建 `ByteArrayKey.forLookup(rawTerm, hash)` 零拷贝方法，只用于 HashMap.get 查找。

**效果：** 100K docs/20K vocab 场景从 4839ms 降到 3951ms（-18%）。

#### 阶段四：new BitSet() 按需扩容

**思考：** 两阶段方案虽然好，但能不能合并成一次扫描？如果用 `new BitSet()`（空构造器）替代 `new BitSet(docCount)`，低频词按需扩容，就不会浪费空间了。

**做法：** 修改 `build()` 使用 `new BitSet()` 替代 `new BitSet(docCount)`。

**效果：** 100K docs/20K vocab 场景从 5.1s 降到 2.6s（2x 加速），内存从 358MB 降到 2.4MB（146x）。

**教训：** 但 `build()` 无法过滤低频词。在大词表 + 随机数据场景（41万唯一词），`build()` 需要 9.3s，而 `buildWithSupportBounds` 只需要 220ms（过滤后只剩 40 个高频词）。

**结论：** `buildWithSupportBounds` + 低频词过滤才是大词表场景的正确路径。`build()` 只在 minSupport=1（不过滤）时适用。

#### 阶段五：OpenHashTable 尝试

**思考：** Java Perf Guru 建议用自定义哈希表替代 HashMap，减少装箱和 equals 开销。

**做法：** 实现 `OpenHashTable.java`，用 `int[]` + `byte[][]` 三个并行数组，开放寻址线性探测。

**结果：** 低频词极多的场景（41万唯一词）下 OpenHashTable 没有优势。HashMap 的桶链已经很短，线性探测在冲突时性能下降更明显。

**教训：** 核心瓶颈不在数据结构，而在低频词数量。**过滤 > 优化**。

#### 阶段六：配置化

**思考：** 老板要求"采样比例不要写死，放到配置文件中"。

**做法：** 将 `SAMPLE_RATIO`、`MIN_SAMPLE_COUNT`、`SAMPLING_SUPPORT_SCALE` 从硬编码常量改为 `EngineTuningConfig` 配置项，并通过 `ExclusiveFrequentItemsetSelector.setXxx()` 方法暴露。

### 最终状态
- 采样质量：完全恢复（Groups diff=0）
- 索引构建：buildWithSupportBounds 过滤低频词，大词表场景 200ms vs 旧版 9.3s
- 配置化：采样参数可通过配置文件调整
- 文档：`optimization-summary-collaborative.md`（改进意见），`worklog-collaborative-2026-04-22.md`（工作日志）

## 2. Architect（架构师） — 工作日志

### 任务
从架构层面分析 fptoken 项目的优化方向，给出优先级建议。

### 思考过程

#### 问题定义
收到 Developer 的瓶颈分析报告：索引构建占 98%。需要从架构角度审视是否还有更好的方案。

#### 关键问题
1. **采样架构是否合理？** 采样在索引构建之后做位过滤，对 98% 的瓶颈无效。但这是正确的——采样本意是优化挖掘阶段，而非索引构建。
2. **两阶段索引构建是否是最优架构？** 先统计频率再分配 BitSet，在大词表场景下效果显著。但两次扫描是否有更好的替代方案？
3. **RoaringBitmap 是否有必要？** 架构上 RoaringBitmap 可以进一步减少稀疏存储内存，但两阶段索引构建 + new BitSet() 按需扩容已经解决了核心问题。老板也确认不需要。

#### 架构决策
1. **P0：两阶段索引构建（已实施）** — 先统计频率，再只对高频词分配 BitSet。内存降低 4-6 倍。
2. **P1：分片并行索引构建** — 百万级文档场景需要。
3. **P1：合并两阶段扫描为单次扫描** — 用 ArrayList<Integer> 记录 docId 列表，扫描结束后统一设位图。但内存与时间的 trade-off 需要评估。

#### 遇到的问题
子 agent 创建失败（HTTP 401: invalid access token or token expired），导致架构分析报告未交付给 Developer。

**解决：** Developer 自行整合了架构分析的发现，确保不遗漏关键建议。

## 3. Java Perf Guru（Java 性能专家） — 工作日志

### 任务
从 JVM 和性能角度分析瓶颈，给出具体的调优建议。

### 思考过程

#### 初步分析
基于 BottleneckProfiler 数据，确认索引构建是唯一瓶颈。深入分析 `build()` 和 `buildWithSupportBounds` 的实现。

#### 问题 1：ByteArrayKey 数组拷贝
**发现：** 两阶段索引的第二次扫描中，`new ByteArrayKey(rawTerm)` 每次拷贝 byte[]。
**解决：** `forLookup()` 静态方法直接引用原始数组，零拷贝。效果：-18%。

#### 问题 2：HashMap vs 自定义哈希表
**思考：** HashMap 有 Entry 对象开销、hash 扰动、equals 虚方法调用。理论上自定义 int[] 开放寻址哈希表应该更快。
**做法：** 实现 OpenHashTable，用三个并行数组（keys[], hashes[], values[]），线性探测。
**结果：** 在大词表场景（41万唯一词）下没有性能优势。分析原因：
- HashMap 桶链已很短（负载因子 0.75）
- 开放寻址的线性探测在冲突时遍历更多的槽
- byte[] 相等比较的开销在两种方案中相同

**教训：** 不要过度优化。HashMap 在当前场景下足够好。

#### 问题 3：GC 分析
**发现：** 两阶段扫描产生大量临时对象。第一次扫描 41 万个 ByteArrayKey + HashMap Entry，第二次扫描 1230 万个 ByteArrayKey.forLookup。
**建议：** 固定堆大小减少 GC 停顿，预分配 HashMap 容量减少 rehash。

#### 最终发现
**核心结论：** 性能瓶颈不在数据结构实现，而在低频词数量。过滤低频词（buildWithSupportBounds）比优化数据结构更有效（10-40x vs 10-15%）。

## 4. QA Lead（质量保证负责人） — 工作日志

### 任务
验证采样质量，确保优化不会引入回归。

### 思考过程

#### 问题 1：采样质量验证
**收到 Developer 的采样策略重构后，第一反应：** 采样结果与全量挖掘结果是否一致？

**做法：** 设计对比测试方案：
1. `SamplingCompareTest` — 随机测试数据，500~10000 文档
2. `LargeScaleTest` — 大规模随机数据，20000~100000 文档
3. `LargeVocabTest` — 大词表确定性数据

**验证指标：** Groups diff（组合数量差异）、doc-refs diff（文档引用差异）

**发现的问题：**
- 初始 20% 采样在大词表下 doc-refs 差异达 -19254
- 根因：低频词在采样集中消失，采样挖掘找不到足够的高频组合

**修复验证：**
- 采样比例提升到 30% + minSupport 缩放 + 全量回算
- 最终结果：Groups diff=0，doc-refs diff=0（三个测试全部通过）

#### 问题 2：采样路径 0 groups 问题
**发现：** 当 minSupport 不缩放时（`samplingSupportScale=1.0`），采样路径挖不到任何组合（0 groups）。

**根因：** 30% 采样集上高频词被稀释，绝对数量达不到 minSupport。

**解决：** 恢复 minSupport 自动缩放（默认按采样比例缩放）。采样集上先用较低阈值挖掘候选，回算时再用全量 minSupport 过滤。

#### 建议
1. 增加自动化质量门禁：每次构建验证 Top-K 组合一致性
2. 增加边界场景测试（minSupport=1、极端分布、极小文档数）
3. 增加采样覆盖率监控指标

## 5. Search Guru（搜索专家） — 工作日志

### 任务
从检索场景角度评估采样策略，确保采样结果在二进制检索中有用。

### 思考过程

#### 问题：采样丢失的组合是否影响检索？
**分析：** 频繁项集用于倒排索引压缩编码。只有高频组合对压缩率有贡献。低频组合即使丢失也不影响检索性能。

**结论：** 采样对检索场景是安全的。当前采样策略（30% + minSupport 缩放）已验证高频组合完全覆盖。

#### 验证建议
**Top-K 高频组合优先验证：** 不需要比较全量 doc-refs，只需要验证 Top-10/20 高频组合是否一致。这比全量比较更高效且更有意义。

#### 配置建议
- 检索场景下推荐采样比例 20-30%（当前 30% 合理）
- 不建议不缩放 minSupport（会导致采样集挖不到组合）

## 6. Big Data Guru（大数据专家） — 工作日志

### 任务
评估当前方案在百万级文档场景下的可扩展性，给出大规模优化建议。

### 思考过程

#### 问题：当前方案能支撑多大规模？
**分析：** 当前 buildWithSupportBounds 是 O(n) 时间复杂度，100K docs 约 2.5s。
- 1M docs 预估：~25s（线性增长）
- 10M docs 预估：~250s（4分钟）
- 内存：1M docs 约 5GB+

**结论：** 百万级文档可以接受，千万级需要优化。

#### 建议
1. **分片并行索引构建（P1）：** ForkJoinPool 分片处理，4 核并行可加速 2-3x
2. **流式索引构建（P2）：** 分批输入，逐步构建，减少内存峰值
3. **分布式索引构建（P3）：** 多机并行，适合千万级以上

## 7. OS Tuning Guru（操作系统调优专家） — 工作日志

### 任务
从操作系统和 JVM 层面分析性能问题，给出系统级调优建议。

### 思考过程

#### 问题：系统资源消耗模式
**分析：**
- CPU：HashMap 操作密集，CPU 计算为主
- 内存：临时对象频繁创建，GC 压力大
- I/O：无（全内存操作）

#### 发现的问题
1. **GC 压力：** 两阶段扫描产生大量临时 ByteArrayKey 和 HashMap Entry 对象。100K docs 下 Young GC 频繁（约 5-10 次/扫描）
2. **HashMap rehash：** 默认初始容量 16，41 万唯一词需要多次 rehash

#### 建议
1. **JVM 参数调优：** `-Xms1g -Xmx1g` 固定堆、`-XX:+UseG1GC`
2. **预分配 HashMap：** `new HashMap<>(expectedVocabSize / 0.75f + 1)`
3. **CPU 亲和性（P3）：** 多路服务器场景
